# Design Specification: Distributed BFV Evaluation Key Generation

**Status**: Design complete — reviewed and corrected (Oracle review applied)  
**Date**: 2026-04-10  
**Scope**: MPC protocol for generating Galois keys and relinearization keys alongside the existing Enclave DKG, without ever reconstructing the full secret key.

---

## 1. Background and Motivation

The auction-bitplane demo currently uses a **shortcut** for evaluation key generation (`build_eval_key_from_committee` in `lib.rs:237–289`): it temporarily reconstructs the full joint secret key `s = s_0 + s_1 + s_2` from all committee members' raw secret keys, builds the evaluation and relinearization keys using fhe.rs's `EvaluationKeyBuilder` and `RelinearizationKey::new()`, then discards the reconstructed key.

This defeats the purpose of the threshold DKG, since the full secret key is materialized — even if briefly — in a single process. A production system must generate these keys **distributedly**, so the joint secret key is never reconstructed.

### What We Need to Generate

1. **Galois keys** for 11 rotations: column rotations by powers of 2 (1, 2, 4, 8, 16, 32, 64, 128, 256, 512) plus one row rotation (exponent `2N − 1 = 4095`).
2. **One relinearization key** for reducing degree-2 ciphertexts (from ct×ct multiplication) back to degree 1.

### How fhe.rs Builds These Keys (Single-Party)

Both key types are built on top of `KeySwitchingKey` (KSK). A KSK switches ciphertexts encrypted under polynomial `from` to ciphertexts encrypted under secret key `s`. Its structure is:

```
For each decomposition index i:
    c1[i] = a_i                              (random polynomial, deterministic from seed)
    c0[i] = e_i − a_i·s + g_i·from          (g_i = RNS garner coefficient)
```

- **Galois key** for exponent k: `KSK(s, σ_k(s))` where `σ_k` is the automorphism `x ↦ x^k`.
- **Relinearization key**: `KSK(s, s²)` where `s²` is computed as `s·s` in NTT domain.

Key code references in fhe.rs (commit `3824c52`):
- `crates/fhe/src/bfv/keys/key_switching_key.rs` — core KSK construction
- `crates/fhe/src/bfv/keys/galois_key.rs` — Galois key = KSK with `from = σ_k(s)`
- `crates/fhe/src/bfv/keys/relinearization_key.rs` — relin key = KSK with `from = s²`
- `crates/fhe/src/bfv/keys/evaluation_key.rs` — builder that manages set of Galois keys

## 2. Mathematical Foundation

### Joint Secret Key

The committee runs DKG to produce shares `s_0, s_1, ..., s_{N-1}` such that the implicit joint secret key is:

```
s = Σ_{j=0}^{N-1} s_j
```

No party ever holds `s`. The joint public key `pk` is computed by aggregating public key shares (existing DKG Phase 5, circuit C5).

**Important distinction**: The DKG produces two types of shares per member:

1. **Additive BFV secret key shares** (`s_j`): Generated in `member_keygen`, used for public key aggregation and eval key generation. These satisfy `s = Σ s_j` exactly.
2. **Shamir threshold shares**: Generated by Shamir-splitting each `s_j` across committee members, used for threshold decryption (any t+1 of N can decrypt).

The eval key MPC protocol described in this document uses **only the additive shares** (`s_j`). It does NOT use the Shamir threshold shares. All N parties must participate (N-out-of-N), matching the additive structure.

### Why Galois Keys Are Easy (Linear)

Galois automorphisms are ring homomorphisms. They distribute over addition:

```
σ_k(s) = σ_k(Σ s_j) = Σ σ_k(s_j)
```

Each party can compute `σ_k(s_j)` locally. The Galois key generation decomposes into independent per-party contributions that are aggregated by addition.

### Why Relinearization Keys Are Hard (Quadratic)

Relinearization requires `s²`:

```
s² = (Σ s_j)² = Σ s_j² + 2·Σ_{i<j} s_i·s_j
```

Each party can compute `s_j²` locally, but the **cross terms** `s_i·s_j` require information from multiple parties. Simple additive aggregation of `s_j²` shares gives the wrong result:

```
Σ s_j² ≠ (Σ s_j)² = s²
```

This non-linearity requires an interactive protocol. The standard approach (Mouchet et al., implemented in Lattigo) uses a 2-round ephemeral-key protocol.

## 3. Protocol Overview

The protocol follows the approach of Mouchet et al. ("Multiparty Homomorphic Encryption from Ring-Learning-with-Errors", POPETS 2021), as implemented in the Lattigo library.

### Design Choices

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Base protocol | Mouchet et al. 2021 | Production-tested in Lattigo, proven security, BFV-compatible |
| Threshold model | N-out-of-N | Matches existing DKG (all 3 members participate); t-out-of-N extension available via Shamir resharing if needed later |
| Adversary model | Semi-honest (passive) | Consistent with existing Enclave threat model |
| CRS mechanism | Seeded PRNG (existing CRP) | The existing DKG already distributes a common random polynomial; same mechanism extended |
| Galois key gen | 1 round, non-interactive | Linearity of automorphisms allows simple aggregation |
| Relin key gen | 2 rounds, interactive | Mandatory due to quadratic non-linearity of s² |

### Communication Rounds Summary

| Phase | Rounds | Messages per Round | Broadcast Size |
|-------|--------|--------------------|---------------|
| CRS setup | 0 (reuse existing CRP) | — | — |
| Galois key gen (per rotation) | 1 | N shares | O(d·L·N) polynomials |
| Relin key gen | 2 | N shares each | O(d·L·N) polynomials |

Where `d` = number of RNS decomposition components, `L` = number of RNS moduli at key level, `N` = committee size (3).

## 4. Protocol Specification: Galois Key Generation

**Goal**: For each required rotation exponent k, produce a `GaloisKey` that switches ciphertexts from `σ_k(s)` to `s`, without any party knowing `s`.

### 4.1 Common Reference String

The current codebase does **not** already distribute a reusable CRP root seed for eval-key generation. It currently samples a single `CommonRandomPoly` for public-key generation. Therefore, eval-key generation needs a **new root-seed distribution step** (or equivalent explicit broadcast of deterministic randomness) after DKG.

From that root seed, all parties derive common random polynomials. For each Galois key exponent k and decomposition index i, derive a common random polynomial:

```
a_{k,i} = PRNG(seed, domain="galois", exponent=k, index=i)
```

This extends the existing common-randomness mechanism, but it is a **new derivation tree**, not simple reuse of the single public-key CRP object already present in the demo. The `c1` component of the KSK is fully determined by the derived randomness — it does not depend on any secret.

### 4.2 Per-Party Share Generation (Local, No Interaction)

Each party j, holding secret key share `s_j`, computes:

```
For each decomposition index i ∈ {0, ..., d-1}:
    σ_k(s_j)  ← substitute(s_j, k)       // polynomial substitution x ↦ x^k
    from_j    ← mod_switch_up(σ_k(s_j))   // promote from ciphertext level to key level
    e_{j,i}   ← sample_small_error()      // fresh noise (at key level)
    share_j[i] = e_{j,i} − a_{k,i}·s_j + g_i·from_j
```

Where `g_i` is the i-th RNS Garner coefficient (same as in `KeySwitchingKey::generate_c0`), and `mod_switch_up` extends the polynomial from ciphertext context (level 0) to key-switching context. This mirrors the single-party `GaloisKey` construction in fhe.rs (`galois_key.rs`), which calls `Poly::mod_switch_to()` on the substituted polynomial before passing it to `KeySwitchingKey::new()`.

**Note on levels**: This protocol currently supports only **level 0** (ciphertext level = 0, ksk level = 0), matching the existing fhe.rs multiparty relin protocol. If multi-level support is needed later, the mod-switch step must target the appropriate key-switching context.

The party broadcasts `{share_j[0], ..., share_j[d-1]}`.

### 4.3 Aggregation (Deterministic Addition)

Any party (or a coordinator) computes:

```
For each decomposition index i:
    c0[i] = Σ_{j=0}^{N-1} share_j[i]
    c1[i] = a_{k,i}                       // from CRS, same for all
```

The aggregated `(c0, c1)` is a valid `KeySwitchingKey` for switching from `σ_k(s)` to `s`.

### 4.4 Correctness Argument

```
c0[i] = Σ_j (e_{j,i} − a_{k,i}·s_j + g_i·from_j)
       = (Σ_j e_{j,i}) − a_{k,i}·(Σ_j s_j) + g_i·(Σ_j from_j)
       = e_agg − a_{k,i}·s + g_i·mod_switch_up(σ_k(s))
```

This has the exact form of a KSK `c0` component with secret key `s` and source polynomial `mod_switch_up(σ_k(s))`, with aggregated error `e_agg = Σ e_{j,i}`. The linearity of `mod_switch_up` and `σ_k` over addition ensures `Σ from_j = mod_switch_up(Σ σ_k(s_j)) = mod_switch_up(σ_k(s))`.

The aggregated error is slightly larger than a single-party error: the **variance** scales by `N`, so the **standard deviation** scales by `√N`. This is acceptable for our parameters.

### 4.5 Integration with Existing DKG

This protocol runs as a **new phase after the existing DKG Phase 5** (public key aggregation, circuit C5). At that point:
- Each party holds their secret key share `s_j`
- The public-key CRP is already distributed
- The eval-key root seed / deterministic CRS description has been distributed
- The joint public key is available

The Galois key generation is purely additive, but it does require **new appended phases** for eval-key CRS distribution and share exchange. It does not require changing the existing five DKG phases themselves.

## 5. Protocol Specification: Relinearization Key Generation

**Goal**: Produce a `RelinearizationKey` that switches ciphertexts from `s²` to `s`, without any party knowing `s`.

This follows the Mouchet et al. 2-round protocol with ephemeral keys. **fhe.rs already implements this exact protocol** in `crate::mbfv::RelinKeyGenerator` (`relin_key_gen.rs`). The specification below documents the protocol for clarity and to define the integration points with Enclave infrastructure; the implementation should use (or closely follow) the existing fhe.rs code.

### 5.1 Intuition

The problem: we need `KSK(s, s²)`, but `s²` cannot be decomposed additively from shares of `s`. The trick is to use ephemeral keys `u_j` (one per party) in a 2-round protocol:

- **Round 1**: Each party publishes RLWE-style encryptions involving `s_j` and `u_j` combined with the CRS. The aggregate `(H0, H1)` encodes `u = Σ u_j` and `s = Σ s_j` jointly.
- **Round 2**: Each party multiplies the Round 1 aggregate by `s_j` (for one component) and `u_j − s_j` (for the other). When aggregated, the `s·u` cross terms cancel in the sum `R0 + R1`, leaving `w_i·s² + noise` — exactly the `s²`-dependent term needed for the KSK.

### 5.2 Round 1: Ephemeral Share Generation

Each party j:

1. **Sample ephemeral secret key** `u_j ← sample_ternary()` (kept private, never broadcast).

2. **Compute Round 1 share**: For each decomposition index i:
   ```
   a_i        ← CRS(seed, domain="relin", index=i)     // common random poly (NTT representation)
   e_{j,i}    ← sample_small_error()
   e'_{j,i}   ← sample_small_error()

   h0_{j}[i]  = −a_i·u_j + w_i·s_j + e_{j,i}
   h1_{j}[i]  =  a_i·s_j + e'_{j,i}
   ```
   Where `w_i` is the i-th RNS **Garner coefficient** (`rns.get_garner(i)` in fhe.rs). This matches both `mbfv/relin_key_gen.rs` and `bfv/keys/key_switching_key.rs`.

3. **Broadcast** `(h0_j, h1_j)` and **retain** `u_j` for Round 2.

### 5.3 Round 1 Aggregation

Aggregate all Round 1 shares:

```
For each decomposition index i:
    H0[i] = Σ_j h0_j[i] = −a_i·u + w_i·s + e_agg
    H1[i] = Σ_j h1_j[i] =  a_i·s + e'_agg
```

Where `u = Σ u_j` and `s = Σ s_j`. Broadcast the aggregate `(H0, H1)` to all parties.

### 5.4 Round 2: Share Computation with Secret Key

Each party j receives the Round 1 aggregate `(H0, H1)` and computes:

```
For each decomposition index i:
    r0_j[i] = s_j · H0[i] + e''_{j,i}
    r1_j[i] = (u_j − s_j) · H1[i] + e'''_{j,i}
```

Note the asymmetry: `r0` uses the **secret key share** `s_j`, while `r1` uses the **difference** `u_j − s_j`. This is the exact protocol implemented in `fhe::mbfv::RelinKeyGenerator::round_2()` (see `relin_key_gen.rs` lines 234–316).

Party j broadcasts `(r0_j, r1_j)`.

### 5.5 Round 2 Aggregation and Key Extraction

Aggregate Round 2 shares:

```
For each decomposition index i:
    R0[i] = Σ_j r0_j[i] = s·H0[i] + noise
           = s·(−a_i·u + w_i·s + e_agg) + noise
           = −a_i·s·u + w_i·s² + noise

    R1[i] = Σ_j r1_j[i] = (u − s)·H1[i] + noise
           = (u − s)·(a_i·s + e'_agg) + noise
           = a_i·s·u − a_i·s² + noise
```

The final relinearization key combines these to cancel the `s·u` cross terms:

```
    c0[i] = R0[i] + R1[i]
           = (−a_i·s·u + w_i·s²) + (a_i·s·u − a_i·s²) + noise
           = (w_i − a_i)·s² + noise

    c1[i] = H1[i] = a_i·s + e'_agg             (from Round 1 aggregate)
```

**Correctness check**: A valid KSK satisfies `c0[i] + c1[i]·s ≈ w_i·from`. Substituting with `from = s²`:

```
    c0[i] + c1[i]·s = (w_i − a_i)·s² + (a_i·s + e')·s + noise
                     = w_i·s² − a_i·s² + a_i·s² + noise
                     = w_i·s² + noise  ✓
```

This is exactly the fhe.rs implementation (`relin_key_gen.rs` lines 343–351): `c0 = Σ r2.h0 + Σ r2.h1`, `c1 = R1_agg.h1`. Before wrapping into `KeySwitchingKey`, both `c0` and `c1` must be converted to `Representation::NttShoup`, matching fhe.rs.

### 5.6 Why This Is Secure

- Each party's ephemeral key `u_j` is never revealed.
- Round 1 shares are RLWE encryptions (masked by `u_j`), computationally hiding `s_j`.
- Round 2 shares use `u_j` with the *aggregate* (not individual shares), so no party learns another's `s_j`.
- Under the RLWE assumption, the protocol is secure against semi-honest adversaries (Mouchet et al. Theorem 1).

### 5.7 Noise Analysis

The aggregated relinearization key has larger noise than a single-party key due to:
1. **Summation of N error samples** in each round (factor of N in error terms).
2. **Products of errors with secret/ephemeral polynomials** in Round 2 (the `s_j · e_agg` and `(u_j − s_j) · e'_agg` terms contribute additional noise).

For our parameters (N=3, ternary secrets with 6×62-bit moduli), the fhe.rs library's `mbfv::RelinKeyGenerator` implements this exact protocol and its test suite (`relinearization_works`) validates correct multiplication and decryption with up to 5 parties. The noise growth is well within the ~370-bit budget.

## 6. Integration with Existing Enclave Infrastructure

### 6.1 Reuse Map

| Existing Component | Reuse for Eval Key Gen | How |
|-------------------|----------------------|-----|
| **Common randomness mechanism** | Root for eval key CRS derivation | Add a new root-seed distribution or deterministic-CRP broadcast, then derive domain-separated randomness for "galois" and "relin" |
| **`SecretKey` coefficients** (`s_j`) | Input to share generation | Already held by each party from DKG |
| **P2P gossip broadcast** | Share distribution | Same broadcast mechanism as DKG shares |
| **`ShareManager` / collectors** | Message collection only | Can be extended to carry eval-key shares, but must operate over additive BFV secret keys, not Shamir-reconstructed decryption material |
| **`ThresholdShareCollector`** | Round management for relin key | Can be extended to support 2-round protocols, but it should not change the underlying additive-share model |
| **Barretenberg ZK backend** | Proof generation for new circuits | Same infrastructure, new circuit definitions |
| **`PublicKeyShare::new_extended()`** | Pattern for share-then-aggregate | Same encrypt-with-error-then-aggregate pattern |
| **Protobuf serialization** | Message format for shares | Extend existing proto definitions |

### 6.2 New Components Required

1. **`GaloisKeyShare`** — per-party share for one Galois key (array of `d` polynomials)
2. **`RelinKeyShareRound1`** — per-party Round 1 share (two arrays of `d` polynomials)
3. **`RelinKeyShareRound2`** — per-party Round 2 share (two arrays of `d` polynomials)
4. **`EvalKeyShareCollector`** — actor that collects shares and triggers aggregation
5. **CRS extension** — domain-separated PRNG expansion for eval key random polynomials

### 6.3 Protocol Sequencing Within DKG

The complete DKG + eval key generation flow:

```
Phase 0:  Generate public-key CRP                      (existing)
Phase 1:  Generate BFV keypairs + PK shares            (existing, C0)
Phase 2:  Shamir share computation                     (existing, C2a/C2b)
Phase 3:  Share encryption + distribution              (existing, C3a/C3b)
Phase 4:  Share decryption verification                (existing, C4a/C4b)
Phase 5:  Public key aggregation                       (existing, C5)
─── NEW PHASES BELOW ───
Phase 6:  Distribute eval-key root seed / deterministic CRS description (NEW)
Phase 7:  Galois key generation (1 round)              (NEW)
Phase 8a: Relin key generation Round 1                 (NEW)
Phase 8b: Relin key generation Round 2                 (NEW)
Phase 9:  Eval key verification                        (NEW, optional)
```

Phases 7 and 8a can potentially run **in parallel** once the eval-key CRS/root seed has been distributed.

## 7. ZK Circuit Requirements

### 7.1 Galois Key Share Correctness (C-GK)

**Purpose**: Prove that a Galois key share was correctly computed from the party's secret key share.

**Public inputs**: `share_j[i]` (the broadcast share), `a_{k,i}` (from CRS), BFV parameters

**Private inputs**: `s_j` (secret key share), `e_{j,i}` (error polynomial)

**Relation**: `share_j[i] = e_{j,i} − a_{k,i}·s_j + g_i·mod_switch_up(σ_k(s_j))` AND `||e_{j,i}||∞ < B_err`

**Reuse potential**: This circuit is structurally identical to the existing **C0 (BFV keypair correctness)** circuit, which proves `pk_j = e_j − a·s_j`. The only difference is:
- C0 proves `pk = e − a·s`
- C-GK proves `share = e − a·s + g·σ_k(s)`

The automorphism `σ_k` is **not just a plain coefficient permutation**. In the negacyclic ring, substitution by an odd exponent can also induce sign flips after reduction modulo `x^N + 1`. The circuit must therefore implement the exact substitution semantics used by `fhe.rs::SubstitutionExponent`, including index remapping and sign handling.

### 7.2 Relin Key Share Correctness — Round 1 (C-RK1)

**Purpose**: Prove Round 1 share was correctly computed.

**Public inputs**: `(h0_j[i], h1_j[i])`, `a_i` (from CRS)

**Private inputs**: `s_j`, `u_j` (ephemeral key), `e_{j,i}`, `e'_{j,i}`

**Relation**:
```
h0_j[i] = −a_i·u_j + w_i·s_j + e_{j,i}
h1_j[i] =  a_i·s_j + e'_{j,i}
||e_{j,i}||∞ < B_err AND ||e'_{j,i}||∞ < B_err AND ||u_j||∞ < B_eph
```

This is more complex than C-GK since it involves two secret polynomials (`s_j` and `u_j`) and two output components.

### 7.3 Relin Key Share Correctness — Round 2 (C-RK2)

**Purpose**: Prove Round 2 share correctly uses the same `u_j` from Round 1.

**Public inputs**: `(r0_j[i], r1_j[i])`, `(H0[i], H1[i])` (Round 1 aggregate)

**Private inputs**: `s_j` (secret key share), `u_j` (same ephemeral key from Round 1), `e''_{j,i}`, `e'''_{j,i}`

**Relation**:
```
r0_j[i] = s_j · H0[i] + e''_{j,i}
r1_j[i] = (u_j − s_j) · H1[i] + e'''_{j,i}
||e||∞ < B_err
```

**Critical**: The circuit must verify that `u_j` is the **same** ephemeral key used in Round 1. This requires either:
- A commitment to `u_j` published in Round 1 (and verified in C-RK2), or
- A combined proof across both rounds (more expensive but more direct)

The commitment approach is recommended: party j publishes `com(u_j) = Hash(u_j)` in Round 1, and C-RK2 verifies the preimage.

### 7.4 Circuit Summary

| Circuit | Based On | Complexity vs C0 | Novel Components |
|---------|----------|-------------------|------------------|
| C-GK | C0 + permutation | ~1.2× C0 | Coefficient permutation |
| C-RK1 | C0 extended | ~2.5× C0 | Two secret inputs, two outputs |
| C-RK2 | New | ~2× C0 | Commitment verification, polynomial multiplication |

### 7.5 Minimum Viable Approach (No ZK)

For a semi-honest adversary model, ZK proofs are **not strictly required** — parties are assumed to follow the protocol honestly. The circuits above are needed only when upgrading to **malicious security**. The protocol can be deployed initially without ZK proofs and upgraded later.

## 8. fhe.rs API Integration

### 8.1 Constructing Keys from Aggregated Shares

fhe.rs does not currently have a complete public API for constructing every eval-key type from pre-computed `(c0, c1)` components. The implementation should keep **Galois keys** and the **relinearization key** separate, matching the actual fhe.rs object model.

**Option A: Proto round-trip for GaloisKey (Preferred — no fork needed)**

`RelinearizationKey::new_from_ksk(ksk)` accepts a pre-built `KeySwitchingKey`, and `KeySwitchingKey` has **`pub` fields**, so the relin key can be constructed directly from aggregated shares:

```rust
// Construct KSK from aggregated c0 and c1
aggregated_c0.iter_mut().for_each(|p| p.change_representation(Representation::NttShoup));
crs_c1.iter_mut().for_each(|p| p.change_representation(Representation::NttShoup));

let ksk = KeySwitchingKey {
    par: params.clone(),
    seed: None,  // No seed — c1 was from CRS, not local PRNG
    c0: aggregated_c0.into_boxed_slice(),
    c1: crs_c1.into_boxed_slice(),
    ciphertext_level: 0,
    ctx_ciphertext: ctx_ciphertext.clone(),
    ksk_level: 0,
    ctx_ksk: ctx_ksk.clone(),
    log_base: 0,
};
let relin_key = RelinearizationKey::new_from_ksk(ksk);
```

For **GaloisKey**, direct struct construction is **not possible** — fields are `pub(crate)`, not `pub`. Two workarounds:

**Option A1: Proto deserialization (no upstream changes)**

Construct a `GaloisKeyProto` from the aggregated KSK and deserialize:

```rust
use fhe::bfv::traits::TryConvertFrom;
use fhe::proto::bfv::{
    EvaluationKey as EvaluationKeyProto,
    GaloisKey as GaloisKeyProto,
    KeySwitchingKey as KeySwitchingKeyProto,
};

// Build the KSK proto from aggregated shares
let ksk_proto = KeySwitchingKeyProto::from(&aggregated_ksk);
let gk_proto = GaloisKeyProto {
    exponent: exponent as u32,
    ksk: Some(ksk_proto),
};
let galois_key = GaloisKey::try_convert_from(&gk_proto, &params)?;
```

This round-trips through protobuf but avoids any fhe.rs fork. The deserialization path is already tested and public.

**Option A2: Upstream patch (cleaner, recommended long-term)**

Submit a small PR to fhe.rs adding:

```rust
impl GaloisKey {
    pub fn new_from_ksk(exponent: usize, ksk: KeySwitchingKey) -> Result<Self> {
        let element = SubstitutionExponent::new(ksk.ctx_ciphertext.clone(), exponent)?;
        Ok(Self { element, ksk })
    }
}
```

This mirrors the existing `RelinearizationKey::new_from_ksk()` pattern. ~5 lines of code.

The `EvaluationKey` is then assembled by proto deserialization from the collection of **GaloisKey protos only**, using the existing `TryConvertFrom<&EvaluationKeyProto>` implementation. The `RelinearizationKey` remains a separate object, so the application-level output stays `(EvaluationKey, RelinearizationKey)`.

**Recommendation**: Option A1 for initial implementation (zero fork), with A2 as a follow-up upstream contribution.

### 8.2 CRS Polynomial Generation

The `c1` components must be identical across all parties. fhe.rs already supports deterministic polynomial generation via `Poly::random_from_seed()`, but the current repo does not yet distribute an eval-key root seed. The implementation must first distribute a new root seed (or an equivalent deterministic CRS description), then derive:

- one **NTT** CRP polynomial per relin decomposition index `i`, and
- one **NttShoup** `c1` polynomial per `(galois exponent, i)` pair.

Conceptually:

```rust
fn crs_polynomial(seed: &[u8; 32], domain: &str, index: usize, repr: Representation) -> Poly {
    let derived_seed = HKDF::expand(seed, &format!("{domain}/{index}"));
    Poly::random_from_seed(ctx, repr, derived_seed)
}
```

The important point is the **separation of derivation domains and representations**; the exact KDF can be chosen to match project conventions.

## 9. Concrete Protocol Walkthrough (3-Party Auction)

### Step-by-step for the auction demo (N=3, generating 11 Galois keys + 1 relin key):

**Setup** (already done by existing DKG):
- Public-key CRP already distributed
- Eval-key root seed / deterministic CRS description distributed
- Each party holds `s_j` (the additive BFV secret key share, not a Shamir reconstruction)
- Joint public key `pk` aggregated

**Galois Key Generation** (11 keys, 1 round):
1. For each rotation exponent k ∈ {3, 9, 81, 2465, 1857, 3713, 3329, 2561, 1025, 2049, 4095} (i.e. `3^rot mod 4096` for rot ∈ {1,2,4,8,16,32,64,128,256,512} plus 4095 for row swap):
   - Party 0: compute `σ_k(s_0)`, mod-switch up, generate share, broadcast
   - Party 1: compute `σ_k(s_1)`, mod-switch up, generate share, broadcast
   - Party 2: compute `σ_k(s_2)`, mod-switch up, generate share, broadcast
   - Coordinator: aggregate 3 shares → `GaloisKey` for rotation k
2. All 11 rotations can be processed in parallel within the single round.

**Relin Key Generation** (1 key, 2 rounds):

*Round 1:*
1. Party 0: sample `u_0`, compute `(h0_0, h1_0)`, broadcast, retain `u_0`
2. Party 1: sample `u_1`, compute `(h0_1, h1_1)`, broadcast, retain `u_1`
3. Party 2: sample `u_2`, compute `(h0_2, h1_2)`, broadcast, retain `u_2`
4. Coordinator: aggregate → `(H0, H1)`, broadcast to all

*Round 2:*
1. Party 0: compute `r0_0 = s_0·H0 + noise`, `r1_0 = (u_0−s_0)·H1 + noise`, broadcast
2. Party 1: compute `r0_1 = s_1·H0 + noise`, `r1_1 = (u_1−s_1)·H1 + noise`, broadcast
3. Party 2: compute `r0_2 = s_2·H0 + noise`, `r1_2 = (u_2−s_2)·H1 + noise`, broadcast
4. Coordinator: `c0 = Σ r0_j + Σ r1_j`, `c1 = H1` → `RelinearizationKey`

**Total communication**: 3 broadcast rounds (1 for Galois, 2 for relin). The Galois round and relin Round 1 can execute in parallel → **2 sequential rounds minimum**.

## 10. Security Considerations

### 10.1 Threat Model

- **Semi-honest (passive)**: All parties follow the protocol but may try to learn others' secrets from observed messages. This matches the existing Enclave threat model.
- **Dishonest minority**: Up to N−1 parties may be corrupted (for N-out-of-N, the protocol is secure as long as at least one party is honest).

### 10.2 Security Properties

1. **Secret key privacy**: No coalition of N−1 parties can recover the full secret key `s`. Each party's share `s_j` remains hidden — broadcast shares are RLWE encryptions.
2. **Ephemeral key privacy** (relin only): Ephemeral keys `u_j` are never broadcast. Only RLWE-masked products appear in messages.
3. **Key indistinguishability**: The distributed key is computationally indistinguishable from a key generated by a single party holding `s` (under RLWE assumption).

### 10.3 Noise Budget Impact

Aggregated keys have larger noise than single-party keys due to summing independent error samples:

- **Galois key noise**: Sum of N independent error samples per decomposition component. For N=3, this increases the error variance by 3× (standard deviation by `√3`) compared to a single-party key. This is well within the noise budget for our parameters (6×62-bit moduli provide ~370 bits of budget).
- **Relin key noise**: Includes additional terms from the ephemeral key interaction in Round 2 (products of errors with ephemeral/secret polynomials). The exact noise growth depends on the coefficient norms of `s_j`, `u_j`, and the number of parties. For our parameters (N=3, ternary secrets), empirical testing with the fhe.rs `mbfv::RelinKeyGenerator` (which implements this exact protocol) confirms successful multiplication and decryption.

**Note**: The smudging parameter λ=80 used in the existing threshold decryption is **separate** from eval key noise — it applies to decryption shares, not key generation. Eval key noise does not require smudging.

### 10.4 Upgrading to Malicious Security

If the threat model is upgraded to malicious (active) adversaries:
1. Add ZK proofs (C-GK, C-RK1, C-RK2) as described in Section 7.
2. Add commitment to ephemeral keys in Round 1 of relin protocol.
3. Verify all proofs before aggregation.

This is a modular upgrade — the protocol structure remains the same.
